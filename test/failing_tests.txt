# --- Yosys abc9 / specify-timing test for Xilinx primitives (not a
# synthesis-equivalence test) ---
# bug3670 instantiates the Xilinx hard primitive RAMB36E1 with deliberately
# missing / under-specified ports.  Its .ys is `read_verilog -lib -specify
# +/xilinx/cells_sim.v; abc9` — it exercises abc9's handling of $setup / clk-to-q
# specify timing on a blackbox, NOT SystemVerilog -> RTLIL conversion.  RAMB36E1
# is undefined without cells_sim.v, so the generic read -> synth -> equiv flow
# fails for BOTH frontends (the Yosys Verilog frontend's own `synth` errors with
# "Module RAMB36E1 ... is not part of the design").  Not a UHDM feature gap.
# (UHDM separately scopes an undefined-module cell as `parent::RAMB36E1` and
# builds a partial blackbox def from one instance's ports — a distinct
# undefined-primitive-blackbox gap, irrelevant to this abc9 test.)
arch/xilinx/bug3670.v

# --- X-semantics / $countbits elaboration test (not a synthesis-equivalence
# test) ---
# mem_bounds checks that out-of-bounds reads are X and counts the X/0/1 bits
# with $countbits ($countbits(mem[-1], 'x) == 8, etc.).  This is a don't-care /
# elaboration-time property: in a synthesis flow X is a DON'T-CARE, not a
# concrete value, so neither frontend can reproduce the bit counts — the Yosys
# Verilog frontend itself folds the X-counting asserts to `assert(1'h0)`.  UHDM
# now returns X for the OOB reads (was a hard "Could not find wire" error), but
# the $countbits asserts still cannot match a wrong reference, so the test is
# not meaningful under read_uhdm -> synth -> equiv.
verilog/mem_bounds.sv
