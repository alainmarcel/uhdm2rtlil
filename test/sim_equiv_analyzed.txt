Test: sva_not
    Same x-init divergence as `sva_counter`: `reg [2:0] cnt;` is
    declared without an init, the sync reset writes 0, but RTL and
    netlist drift apart on random `ping`/`cfg` stimulus because
    Verilator's `'x` propagation in the `cnt - |cnt` arithmetic
    differs from the gate-level netlist's deterministic-0 behaviour.
    Same diagnosis: known x-init divergence, the SVA `not (...[*N])`
    support is the actual subject of the test and is validated via
    synth.

Test: sva_counter
    Counter DUT with non-initialised `reg [7:0] cnt` and synchronous
    reset (`if (reset) cnt <= 0`).  After the 5-cycle reset hold the
    counter should be 0 in both forms, but Verilator's RTL and the
    Yosys-synthesised `$_SDFFE_PP0P_` netlist diverge on random
    up/down stimulus — netlist increments cleanly while RTL drifts,
    consistent with Verilator preserving `'x` in some upper bits at
    init while the gate-level synch reset legalises to 0.  Adding
    `reg [7:0] cnt = 0` would close the gap but changes the source's
    "no-init synchronous reset" intent (which is the SVA test's
    point).  Recorded as a known x-init divergence; the SVA itself
    is still validated through the synth path.

Test: unbased_unsized_param
    The test's whole purpose is to exercise the unbased-unsized fill
    literals `'0` / `'1` / `'x` / `'z` in localparam, continuous-
    assign, port-connection, and module-parameter contexts.  Twelve
    of the sixteen observable signals deliberately carry `'x` or `'z`
    values (`o03`, `o04`, `o07`, `o08`, `o11`, `o12`, `l03`, `l04`,
    `o15`, `o16`, and bits of the 3'-cast forms).
    RTL preserves the four-state values; synth legalises both `'x`
    and `'z` to `'0` (Yosys is x-pessimistic and has no Z model).
    Co-sim therefore mismatches every cycle on the x/z signals.  This
    is fundamental and not a UHDM-frontend bug.

Test: sva_value_change_sim
    The DUT exercises SVA `$rose` / `$fell` / `$stable` on registers
    that are deliberately set to `'x` in some case-FSM states (e.g.
    `c <= 'x` at counter==1, 3; `a <= 'x; wide_a <= 'x` at counter==4).
    RTL simulation in Verilator preserves the `'x` bits per IEEE 1800
    semantics, but the synthesized gate-level netlist has no notion of
    `'x` — Yosys's synth pass legalises those assignments to a
    deterministic 0 or 1.  Co-sim then mismatches on `c_o` and `a_o`
    whenever the FSM is in an `'x`-write state.
    This is a fundamental RTL-vs-synth divergence (x-propagation), not
    a UHDM-frontend bug; the test's intent is to exercise SVA over
    `'x` values, which co-sim is the wrong tool to validate.

Test: jeras_tcb_gpio
    Imported from jeras/UHDM-tests' tcb_if + tcb_gpio + tcb_gpio_wrap.
    The test exercises modport-port handling (`tcb_if.sub bus` passed
    to `tcb_gpio`).  Earlier PRs fixed the modport-signal access
    (`bus.adr[3:0]` part-select, `bus.rdt <=` LHS write); this PR
    fixes the *interface body* path:
    (1) `interface.cpp` builds a `<sig> → \<iface>.<sig>` mapping for
        all the interface's `Variables()` / `Nets()` and passes it as
        `input_mapping` to `import_expression` for each cont_assign
        RHS, so `assign trn = vld & rdy` lowers to
        `\bus.trn = \bus.vld AND \bus.rdy` instead of writing to raw
        `\vld` / `\rdy` placeholders.
    (2) The port loop now creates the per-signal wire when it doesn't
        already exist — interface ports like `(input logic clk, rst)`
        aren't in `Variables()`/`Nets()`, so the earlier code left
        `\bus.clk` / `\bus.rst` un-allocated and the downstream
        `tcb_gpio` instance had no port to connect those to.
        Result: in the synth output, the gpio cell now gets
        `.\bus.clk (clk)` and `.\bus.rst (rst)` (was silently missing).
    With both fixes, the netlist's interface signals (`bus.trn`,
    `bus.idl`, `bus.clk`, `bus.rst`) are all driven and the FFs
    inside `tcb_gpio` actually clock.  Cycles 115-117 and 181 (the
    original mismatch set, all from bus_rdt / gpio_e being undriven)
    now agree between RTL and netlist.
    The residual ~14 mismatches at the tail of the 200-cycle co-sim
    (cycles 184-199, `bus_rdt rtl=0 nl=<held value>`) are a different
    behavioural pattern: Verilator's RTL simulation re-fires the
    `always_ff @(posedge bus.clk, posedge bus.rst)` on each `rst`
    edge transition delivered by the random testbench, while the
    synthesised `$_DFFE_PP0P_` cell has a *level-sensitive* `R` pin.
    With random per-cycle `rst` flips the two evaluators sample the
    reset window differently — a known cosim/synth artifact when a
    testbench drives async-reset DUTs without a clean reset hold.
    Not a UHDM-frontend bug; the underlying elaboration is now
    correct.

Test: fib_simple
    CONFIRMED ARTIFACT (not a UHDM bug).  An `initial` block computes an
    output from a RUNTIME input: `initial fib2 = off + fib(2);` (= off+1).
    Verilator runs the initial once at t=0 and LATCHES off+1 (off=0 -> 1,
    held); synth makes it COMBINATIONAL (fib2 = off+1, tracks off).  Both
    UHDM and Verilog frontends produce the combinational form (verified:
    both netlists diverge from RTL identically), so this is inherent to
    initial-reads-input under co-sim, not a frontend bug.

Test: fsm
    CONFIRMED ARTIFACT (not a UHDM bug).  The UHDM and Verilog netlists are
    structurally identical (same switch/case tree, assignments, sync rule);
    only auto-generated cell names differ.  The co-sim mismatch is an
    X-init / `#1` intra-assignment-delay (Verilator --timing) artifact that
    affects both frontends equally.

Test: full_case_latch
    CONFIRMED ARTIFACT (not a UHDM bug).  `(* full_case *)` case covering
    sel 0..2; sel==3 is unreachable/don't-care.  Both UHDM and Verilog
    frontends honor full_case (0 latches) and produce the IDENTICAL sel==3
    divergence; Verilator does NOT honor the pragma and infers a latch
    (holds the prior sel==2 value).  Mismatch is only at sel==3.

Test: rp32_r5p_alu
    CONFIRMED ARTIFACT (not a UHDM bug) — design don't-cares.  r5p_alu takes the
    decoded-instruction struct `dec` as a port; this test drives a random 32-bit
    instruction through the design's own decoder dec32() (wrap.sv) so `dec` is
    always valid.  For VALID ALU ops the UHDM netlist is now correct — directed
    eval of an ADD (rs1=5, rs2=3) gives rd=8, sum=8 (it gave rd=5 before the
    dec32 struct-returning-function fix).  The residual co-sim mismatch is the
    design's intentional `'x` don't-cares: for non-ALU opcodes r5p_alu sets
    add_inv/add_uns = 1'bx (and mux_op1/op2 = 'x in the default case), so the RTL
    output is X (Verilator maps X->0) while the synthesized netlist fills the
    don't-care with a concrete value.  The upstream README documents this:
    "X propagation is not simulated well, so I had to run regression tests with
    mapping X separately to 0 and 1".  Verilog frontend can't synthesize the SV
    packages, so no UHDM-vs-Verilog miter is possible (uhdm-only).

Test: rp32_r5p_wbu
    CONFIRMED ARTIFACT (not a UHDM bug) — design don't-cares.  r5p_wbu takes the
    decoded-instruction struct `dec` as a port; this test drives a random 32-bit
    instruction through dec32() (wrap.sv) so `dec` is always valid.  r5p_wbu's
    registered outputs (wen/adr/dat) are the GPR write-back selected by dec.opc;
    for non-write-back instructions wen=0 and dat is a don't-care, so the RTL
    output is X (Verilator maps X->0) while the synthesized netlist fills it with
    a concrete value.  Same class as rp32_r5p_alu (and the README X-prop note).
    uhdm-only (Verilog frontend can't synthesize the SV packages).

Test: rp32_r5p_mdu
    CONFIRMED ARTIFACT (not a UHDM bug) — design don't-cares.  r5p_mdu takes the
    decoded-instruction struct `ctl` as a port and only drives `rd` for M-extension
    (mul/div/rem) opcodes; for every other instruction `rd` is a don't-care, so
    the random-`ctl` co-sim mismatches on X (RTL) vs a concrete netlist value.
    Same class as rp32_r5p_alu/r5p_wbu.  (A dec32 wrapper makes the synthesized
    netlist keep a parameterized $paramod\r5p_mdu that Verilator can't read, so
    this one stays bare.)  uhdm-only.

Test: rp32_r5p_mouse
    CONFIRMED ARTIFACT (not a UHDM bug) — Verilator X-propagation on a flat
    RISC-V CPU core.  The co-sim matches for ~191 cycles, then the Verilator RTL
    goes idle (all tcb_* outputs = 0) at cycle ~192 while the synthesized netlist
    keeps fetching.  Adjudicated: the Yosys-VERILOG-frontend netlist diverges
    from the same Verilator RTL at the SAME cycle 192 (rtl=0, netlist active), so
    the divergence is Verilator's poor X-propagation (the upstream README's
    documented caveat: "X propagation is not simulated well ... map X to 0 and 1
    separately"), affecting both frontends equally — not a UHDM bug.  The module
    SYNTHESISES cleanly via UHDM (after the DEC:x and for-loop-in-function fixes).
    The UHDM-vs-Verilog formal equiv is a separate reset-dependent false-positive
    (see sim_equiv_classification.txt) and is tracked in failing_tests.txt.
